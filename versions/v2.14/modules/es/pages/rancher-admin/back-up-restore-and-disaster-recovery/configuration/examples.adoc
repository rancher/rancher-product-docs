Esta sección contiene ejemplos de recursos personalizados de copia de seguridad y restauración.

La ubicación de almacenamiento predeterminada de las copias de seguridad se configura al instalar o actualizar versión `rancher-backup`el operador.

Las copias de seguridad cifradas solo se pueden restaurar si el recurso personalizado de restauración utiliza el mismo secreto de configuración de cifrado que se utilizó para crear la copia de seguridad.

[#_backup]
== Copia de seguridad

Esta sección contiene ejemplos de recursos personalizados de copia de seguridad.

NOTE: Consulta la {xref-rancher-backup-config}[página de referencia de configuración de copia de seguridad] para más información sobre cómo configurar las opciones a continuación.

[#_backup_in_the_default_location_with_encryption]
=== Copia de seguridad en la ubicación predeterminada con cifrado

[,yaml]
----
apiVersion: resources.cattle.io/v1
kind: Backup
metadata:
  name: default-location-encrypted-backup
spec:
  resourceSetName: rancher-resource-set-full
  encryptionConfigSecretName: encryptionconfig
----

[#_recurring_backup_in_the_default_location]
=== Copia de seguridad recurrente en la ubicación predeterminada

[,yaml]
----
apiVersion: resources.cattle.io/v1
kind: Backup
metadata:
  name: default-location-recurring-backup
spec:
  resourceSetName: rancher-resource-set-basic
  schedule: "@every 1h"
  retentionCount: 10
----

=== Copia de seguridad cifrada en Minio

[,yaml]
----
apiVersion: resources.cattle.io/v1
kind: Backup
metadata:
  name: minio-backup
spec:
  storageLocation:
    s3:
      credentialSecretName: minio-creds

[#_backup_from_ec2_nodes_with_iam_permission_to_access_s3]
=== Copia de seguridad desde nodos EC2 con permiso IAM para acceder a S3

Este ejemplo muestra que no es necesario proporcionar el secreto de credenciales de AWS para crear una copia de seguridad si los nodos que ejecutan `rancher-backup` tienen {xref-rancher-backup-config}#_iam_permissions_for_ec2_nodes_to_access_s3[estos permisos para acceder a S3.]

[,yaml]
----
apiVersion: resources.cattle.io/v1
kind: Backup
metadata:
  name: s3-iam-backup
spec:
  storageLocation:
    s3:
      bucketName: rancher-backups
      folder: ecm1
      region: us-west-2
      endpoint: s3.us-west-2.amazonaws.com
  resourceSetName: rancher-resource-set-full
  encryptionConfigSecretName: encryptionconfig
----

#  encryptionConfigSecretName: test-encryptionconfig

[#_example_encryptionconfiguration]
== Ejemplo de configuración de encriptación

Los fragmentos a continuación demuestran dos tipos diferentes de secretos y su relevancia con respecto a la copia de seguridad y restauración de recursos personalizados. La creación del secreto se puede realizar con el siguiente comando:

[,console]
----
kubectl create secret generic example-encryptionconfig \
  --from-file=./encryption-provider-config.yaml \
  -n cattle-resources-system
----

El primer ejemplo es el de un secreto utilizado para cifrar los archivos de copia de seguridad. El operador de copia de seguridad leerá el contenido de la clave **encryption-provider-config.yaml**, que contiene la definición de un recurso de configuración de cifrado codificado como Base64.

El segundo ejemplo es el del recurso de configuración de cifrado de Kubernetes en sí, siendo la forma en texto plano del contenido codificado en Base64 del primer ejemplo. Este recurso también se utiliza para cifrar secretos cuando se almacenan en etcd y en casos generales de cifrado en reposo en Kubernetes. Más información sobre eso se puede encontrar en la https://kubernetes.io/docs/tasks/administer-cluster/encrypt-data/[documentación de sentido ascendente]. Las claves definidas en este recurso son esenciales para acceder al contenido de la copia de seguridad encriptada, particularmente durante el proceso de restauración. Por esa razón, estas deben mantenerse seguras, confidenciales y fácilmente recuperables.
