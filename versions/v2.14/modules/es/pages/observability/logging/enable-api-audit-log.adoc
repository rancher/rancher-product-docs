Puedes habilitar el registro de auditoría de la API para registrar la secuencia de eventos del sistema iniciados por usuarios individuales. Puedes saber qué ocurrió, cuándo ocurrió, quién lo inició y a qué clúster se vio afectado. Cuando habilitas esta función, todas las solicitudes a la API de Rancher y todas sus respuestas se escriben en un registro.

Puedes habilitar la auditoría de la API durante la instalación o durante la actualización de Rancher.

[#_enabling_api_audit_log]
== Habilitar el registro de auditoría de la API

El registro de auditoría está habilitado y configurado pasando variables de entorno al contenedor del servidor de Rancher. Consulta lo siguiente para habilitarlo en tu instalación.

* {xref-helm-chart-options}#_api_audit_log[Kubernetes Install]

[#_api_audit_log_options]
== Opciones de registro de auditoría de la API

El uso a continuación define reglas sobre lo que el registro de auditoría debe registrar y qué datos debe incluir:

|===
| Parámetro | Descripción

| `AUDIT_LOG_ENABLED`
| `false` - Desactiva el registro de auditoría (configuración predeterminada). +
 +
`true` - Habilita el registro de auditoría.

| `AUDIT_LEVEL`
| `0` - Registra metadatos de solicitud y respuesta (configuración predeterminada). +
`1` - Registra encabezados de solicitud y respuesta. +
`2` - Registra el cuerpo de la solicitud +
`3` - Registra el cuerpo de la respuesta. Cada nivel de registro es acumulativo y cada nivel subsiguiente registra los datos del nivel anterior. Cada transacción de registro para un par de solicitud/respuesta utiliza el mismo valor `auditID`. +
 +
Consulta <<_audit_log_levels,Registro de Nivel de Auditoría>> para ver una tabla que muestra qué registra cada ajuste.

| `AUDIT_LOG_PATH`
| Ruta del registro de auditoría para la API del Servidor Rancher. La ruta por defecto del registro de auditoría es `/var/log/auditlog/rancher-api-audit.log`. Puedes montar el directorio del registro de auditoría en el host. +
 +
Ejemplo de Uso: `AUDIT_LOG_PATH=/my/custom/path/`



| `AUDIT_LOG_MAXAGE`
| Define el número máximo de días para conservar los antiguos archivos de registro de auditoría. El valor por defecto es de 10 días.

| `AUDIT_LOG_MAXBACKUP`
| Define el número máximo de archivos de registro de auditoría a conservar. El valor por defecto es 10.

| `AUDIT_LOG_MAXSIZE`
| Define el tamaño máximo en megabytes del archivo de registro de auditoría antes de que se rote. El tamaño por defecto es de 100M.
|===

[#_audit_log_policies]
== Políticas de Registro de Auditoría

Las políticas de registro de auditoría permiten a los usuarios finales configurar redacciones utilizando `AuditPolicy` CRs de ámbito de clúster además de las <<Default redactions & filters,redacciones y filtros predeterminados>>.

Todas las políticas de registro de auditoría configuradas son aditivas.

Las políticas de redacción para los encabezados utilizan un motor de expresiones regulares (regex) para redactar encabezados, mientras que un motor JSONPath se utiliza para redactar los encabezados de solicitud/respuesta.

El motor JSONPath no soporta las expresiones de script o filtro. Para comenzar con las expresiones JSONPath, un buen recurso a consultar es https://goessner.net/articles/JsonPath/[el artículo de Stafan Goessner sobre JSONPath].

La estructura de un CR de política de auditoría es la siguiente:

[,yaml]
----
apiVersion: auditlog.cattle.io/v1
kind: AuditPolicy
spec:
  enabled : true # true/false
  # list of API request filters
  filters:
    - action: allow # allow/deny
        # would allow logs sent to "/foo/some/endpoint" but not "/foo" or "/foobar".
      requestURI: "/foo/.*"
  # additionalRedactions allows configuration of redactions on headers using `jsonpath` expressions
  additionalRedactions:
    # redacts headers based on regex expressions
    - headers:
      - "Cache.*"
      # paths redacts information from request and response bodies based on json path expressions
      paths: 
      - "$.gitCommit"
  verbosity:
    level : 0 # matches the levels in the above audit log table
    # request allows fine-grained control over which request data 
    # gets included. This overrides the behaviour of the generic verbosity.level
    request:
      headers : true # true/false
      body : true # true/false
    # response allows fine-grained control over which response data 
    # gets included. This overrides the behaviour of the generic verbosity.level
    response:
      headers : true # true/false
      body: true # true/false
----

[#_examples]
=== Ejemplos

El siguiente ejemplo muestra solo las solicitudes de registro que contienen `login` en la vía de solicitud al registro de auditoría:

[,yaml]
----
apiVersion: auditlog.cattle.io/v1
kind: AuditPolicy
metadata:
  name: filters
spec:
  filters:
    - action: deny
      requestUri: ".*"
    - action: allow
      requestUri: ".*login.*"
----

El siguiente ejemplo muestra cómo redactar campos específicos que contienen `gitCommint` en los cuerpos de solicitud/respuesta:

[,yaml]
----
apiVersion: auditlog.cattle.io/v1
kind: AuditPolicy
metadata:
  name: redactions
spec:
  additionalRedactions:
    - paths:
      - "$.gitCommit"
----

[#_default_redactions_filters]
=== Redacciones y filtros predeterminados

El controlador de registro de auditoría viene con redacciones integradas por defecto para información sensible común.

[#_redacted_headers]
==== Encabezados redactados

Encabezados genéricos:

* `Cookie`
* `Set-Cookie`
* `X-Api-Set-Cookie-Header`
* `Authorization`
* `X-Api-Tunnel-Params`
* `X-Api-Tunnel-Token`
* `X-Api-Auth-Header`
* `X-Amz-Security-Token`

[#_redacted_body_fields]
==== Campos del cuerpo redactados

Campos genéricos del cuerpo:

* `credentials`
* `applicationSecret`
* `oauthCredential`
* `serviceAccountCredential`
* `spKey`
* `spCert`
* `certificate`
* `privateKey`
* `secretsEncryptionConfig`
* `manifestUrl`
* `insecureWindowsNodeCommand`
* `insecureNodeCommand`
* `insecureCommand`
* `command`
* `nodeCommand`
* `windowsNodeCommand`
* `clientRandom`

Redactor de regex del cuerpo genérico:

- `".*([pP]assword|[Kk]ube[Cc]onfig|[Tt]oken).*"`

[#_cluster_driver]
==== Controlador de clúster

Por defecto, cualquier solicitud de la API con campos vinculados a controladores de clúster tendrá cualquier campo que no sea `public*` o `optional*` redactado por el controlador de registro de auditoría.

[#_redacted_uris]
==== URIs redactados

Cualquier punto final que contenga `secrets` o `configmaps` tendrá campos relevantes eliminados de los cuerpos de solicitud y respuesta. Además, cualquier punto final que contenga `/v3/imports/*` tendrá su URI redactado.

== Visualización de los registros de auditoría de la API

[#_docker_install]
=== Instalación de Docker

Comparte el directorio `AUDIT_LOG_PATH` (Predeterminado: `/var/log/auditlog`) con el sistema anfitrión. El registro puede ser analizado por herramientas CLI estándar o enviado a una herramienta de recopilación de registros como Fluentd, Filebeat, Logstash, etc.

[#_kubernetes_install]
=== Instalación de Kubernetes

Habilitar el registro de auditoría de la API con la instalación del gráfico de Helm creará un contenedor sidecar `rancher-audit-log` en el pod de Rancher. Este contenedor transmitirá el registro a la salida estándar (stdout). Puedes ver el registro como lo harías con cualquier registro de contenedor.

El contenedor `rancher-audit-log` es parte del pod `rancher` en el espacio de nombres `cattle-system`.

[#_cli]
==== CLI

[,bash]
----
kubectl -n cattle-system logs -f rancher-84d886bdbb-s4s69 rancher-audit-log
----
