El SIG de Kubernetes y el Comité de Respuesta de Seguridad anunciaron la https://kubernetes.io/blog/2025/11/11/ingress-nginx-retirement/[jubilación del proyecto Ingress NGINX]. El mantenimiento de mejor esfuerzo por parte de upstream, y todos los lanzamientos, correcciones de fallos o actualizaciones de seguridad asociadas, finalizaron en marzo de 2026.

Para apoyar a los usuarios durante esta transición, 

ifeval::["{build-type}" != "community"]
SUSE proporciona plazos de soporte extendido y rutas de migración claras a Traefik. 
endif::[]
ifeval::["{build-type}" == "community"]
Rancher proporciona rutas de migración claras a Traefik.
endif::[]

Esta guía centraliza la información sobre cómo proceder según tu escenario de ampliación específico.

[#_support_and_timelines]
== Soporte y Plazos

ifeval::["{build-type}" != "community"]
Para organizaciones que no pueden migrar de inmediato, {rancher-product-name} LTS proporciona soporte extendido para la versión 1.35 de RKE2. 

[IMPORTANT]
.Soporte extendido para Ingress NGINX
====
{rancher-product-name} LTS proporciona parches para vulnerabilidades y exposiciones comunes (CVEs) críticas e importantes para el componente Ingress NGINX hasta diciembre de 2027. No se añadirán nuevas características durante este período.
====
endif::[]

ifeval::["{build-type}" == "community"]
Los ciclos de vida de Rancher y RKE2 se alinean con el calendario de jubilación de upstream.

[IMPORTANT]
====
Después de marzo de 2026, las imágenes de Ingress NGINX de upstream ya no recibirán actualizaciones. Cualquier imagen construida con parches después de esta fecha está restringida a clientes comerciales. Los usuarios deben migrar a Traefik u otro controlador de Ingress soportado antes de esta fecha límite para garantizar la seguridad y la compatibilidad continuas.
====
endif::[]

[#_migration_paths_by_environment]
== Rutas de migración por entorno

ifeval::["{build-type}" != "community"]
Para las organizaciones listas para migrar, SUSE ofrece una ruta de migración soportada hacia Traefik. 
endif::[]
ifeval::["{build-type}" == "community"]
Los ciclos de vida de Rancher y RKE2 se alinean con el calendario de jubilación de upstream.
endif::[]

Traefik incluye una capa de compatibilidad que puede interpretar muchas anotaciones existentes de Ingress NGINX. Identifica tu entorno de clúster a continuación para encontrar el enfoque de migración correcto.

[#_standalone_rke2_clusters]
=== Clústeres RKE2 independientes

Para clústeres RKE2 independientes o importados que actualmente utilizan Ingress NGINX, sigue la https://support.scc.suse.com/s/kb/Migrating-from-Ingress-NGINX-to-Traefik-in-a-standalone-RKE2-cluster?language=en_US[guía de migración oficial para clústeres RKE2 independientes]. 

El proceso de migración implica una estrategia de cuatro fases:

. *Configuración de controlador de ingreso dual:* Habilita Traefik junto a Ingress NGINX utilizando puertos temporales y no conflictivos.
. *Migración y validación paralelas:* Duplica los recursos de Ingress y prueba cómo gestiona Traefik las anotaciones existentes.
. *Cambio final:* Elimina Ingress NGINX y configura Traefik para usar los puertos estándar.
. *Limpieza:* Elimina los objetos de Ingress heredados.

[#_rancher_server_on_rke2_local_clusters]
=== Servidor Rancher en RKE2 (clústeres locales)

Al migrar un clúster local de Rancher, el recurso de Ingress de Rancher requiere un manejo específico para prevenir bloqueos. Sigue la https://support.scc.suse.com/s/kb/How-to-migrate-the-Rancher-Ingress-to-Traefik-in-an-RKE2-cluster?language=en_US[guía específica para migrar el Ingress de Rancher a Traefik en un clúster RKE2]. Esta guía se basa en las fases de migración independientes pero incluye pasos adaptados al servidor de gestión de Rancher.

[#_downstream_rke2_clusters_provisioned_by_rancher]
=== Clústeres RKE2 descendentes (provisionados por Rancher)

Para clústeres RKE2 provisionados y gestionados por {rancher-product-name}, las opciones de migración están integradas directamente en la interfaz de usuario.

[NOTE]
====
ifeval::["{build-type}" != "community"]
Esta opción de migración solo es posible en Rancher v2.13.5+ y v2.14.
endif::[]
ifeval::["{build-type}" == "community"]
Esta opción de migración solo es posible en Rancher v2.14.0+.
endif::[]
====

La interfaz de configuración del clúster proporciona una opción de migración en Modo Dual. Esto te permite probar y migrar el tráfico de Ingress NGINX a Traefik de forma segura directamente desde la pantalla de gestión del clúster.

[#_rancher_on_managed_kubernetes_amazon_eks_azure_aks_google_gke]
=== Rancher en Kubernetes gestionado (Amazon EKS, Azure AKS, Google GKE)

Si ejecutas {rancher-product-name} en un servicio de Kubernetes gestionado como {xref-rancher-on-amazon-eks}[Amazon Elastic Kubernetes Service (EKS)], {xref-rancher-on-aks}[Azure Kubernetes Service (AKS)], o {xref-rancher-on-gke}[Google Kubernetes Engine (GKE)], la recomendación es migrar a Traefik. 

ifeval::["{build-type}" != "community"]
Los clientes de {rancher-product-name} pueden aprovechar la Colección de Aplicaciones para desplegar y gestionar el proxy Traefik.
endif::[]
ifeval::["{build-type}" == "community"]
Los usuarios deben desplegar y migrar a la distribución del proxy Traefik upstream.
endif::[]

de las siguientes maneras:

* {longhorn-product-name}: No hay impacto en el backend de Longhorn. Sin embargo, los administradores deben reconfigurar su Ingress para actualizar la versión de la interfaz de usuario de Longhorn. Para más información, consulta {url-storage-longhorn-ingress-traefik}[Crear un Ingress con Autenticación Básica Traefik].
* {fleet-product-name}: La configuración de un servicio webhook se ve afectada. Consulta la {url-continuous-delivery-webhook}[documentación de Fleet] para más información.
* Se han eliminado las referencias de Ingress NGINX en la documentación de otros proyectos.
