= Referenz für Ressourcenkontingenttypen

:revdate: 2026-04-08

:product-path: cluster-admin/project-admin/project-resource-quotas/resource-quota-types.adoc

Wenn Sie ein Ressourcenkontingent erstellen, konfigurieren Sie den Pool von Ressourcen, die dem Projekt zur Verfügung stehen. Rancher unterstützt die Verwendung beliebiger Ressourcenreferenzen und deren Kontingente. Dies ermöglicht es Ihnen, alle Upstream https://kubernetes.io/docs/concepts/policy/resource-quotas/#types-of-resource-quota[Kubernetes `ResourceQuota`] Typen bei der Verwaltung von Projektressourcenkontingenten zu nutzen.

Sie können Ressourcengrenzen für die folgenden vordefinierten Ressourcentypen festlegen, wobei der `Custom` Typ die Spezifikation beliebiger Ressourcen und deren Kontingente ermöglicht.

[NOTE]
====
Die Unterstützung für beliebige Ressourcenreferenzen mit dem `Custom` Typ umfasst keine Ressourcen in der `ext.cattle.io` API-Gruppe.
====

|===
| Ressourcentyp | Beschreibung

| CPU Limit*
| Die maximale Menge an CPU (in https://kubernetes.io/docs/concepts/configuration/manage-compute-resources-container/#meaning-of-cpu[Millicores]), die dem Projekt/Namensraum zugewiesen ist.^1^

| CPU-Reservierung*
| Die minimale Menge an CPU (in Millicores), die dem Projekt/Namensraum garantiert wird.^1^

| Speicherlimit*
| Die maximale Menge an Speicher (in https://kubernetes.io/docs/concepts/configuration/manage-resources-containers/#meaning-of-memory[Bytes]), die dem Projekt/Namensraum zugewiesen ist.^1^

| Speicherreservierung*
| Die minimale Menge an Speicher (in Bytes), die dem Projekt/Namensraum garantiert wird.^1^

| Speicherreservierung
| Die minimale Menge an Speicher (in Gigabyte), die dem Projekt/Namensraum garantiert wird.

| Lastenausgleichsdienste
| Die maximale Anzahl von Lastenausgleichsdiensten, die im Projekt/Namensraum existieren können.

| Node-Port-Dienste
| Die maximale Anzahl von Node-Port-Diensten, die im Projekt/Namensraum existieren können.

| Pods
| Die maximale Anzahl von Pods, die im Projekt/Namensraum in einem nicht-terminalen Zustand existieren können (d.h. Pods, bei denen der `.status.phase in (Failed, Succeeded)`-Wert true ist).

| Services
| Die maximale Anzahl von Diensten, die im Projekt/Namensraum existieren können.

| ConfigMaps
| Die maximale Anzahl von ConfigMaps, die im Projekt/Namensraum existieren können.

| Persistente Volume Claims
| Die maximale Anzahl von persistenten Volume Claims, die im Projekt/Namensraum existieren können.

| Replikationscontroller
| Die maximale Anzahl von Replikationscontrollern, die im Projekt/Namensraum existieren können.

| Geheimnisse
| Die maximale Anzahl von Secrets, die im Projekt/Namensraum existieren können.

| Benutzerdefiniert**
| Die Spezifikation von beliebigen Ressourcen und deren Kontingente, über die in Projekten integrierten Ressourcentypen hinaus, wie oben aufgeführt.
|===

[NOTE]
.*^*^*
====
Beim Festlegen von Ressourcenquoten sollten Sie, wenn Sie auf einem Projekt oder Namespace Einstellungen bezüglich CPU oder Speicher (d.h. Limits oder Reservierungen) vornehmen, sicherstellen, dass bei der Erstellung jedes Containers das entsprechende CPU- oder Speicherfeld gesetzt wird. Ein Standardressourcenlimit für Container kann gleichzeitig festgelegt werden, um die Notwendigkeit zu vermeiden, diese Limits für jede Arbeitslast explizit festzulegen. Siehe die https://kubernetes.io/docs/concepts/policy/resource-quotas/#requests-vs-limits[Kubernetes-Dokumentation] für weitere Details, warum dies erforderlich ist.
====

[NOTE]
.*^*\^**^*^*
====
Beispiel:

* `requests.nvidia.com/gpu: 4`
* `gold.storageclass.storage.k8s.io/requests.storage: 500Gi`
* `count/podtemplates: 10`

Siehe die https://kubernetes.io/docs/concepts/policy/resource-quotas/#quota-for-extended-resources[Kubernetes-Dokumentation] für viele weitere Beispiele.
====
