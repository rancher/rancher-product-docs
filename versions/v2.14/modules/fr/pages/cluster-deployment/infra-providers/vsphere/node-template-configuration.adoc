[#_account_access]
== Accès au compte

[cols="<,^,<"]
|===
| Paramètre | Requis | Description

| Identifiants Cloud
| *
| Vos informations d'accès à votre compte vSphere, stockées dans un {xref-cloud-credentials}[identifiant cloud].
|===

Votre identifiant cloud contient les champs suivants{nbsp}:

|===
| Champ d'identifiant{nbsp}: | Description

| vCenter or ESXi Server
| Entrez le nom d'hôte/IP du vCenter ou de l'ESXi. ESXi est la plate-forme de virtualisation où vous créez et exécutez des machines virtuelles et des applicatifs virtuels. vCenter Server est le service par lequel vous gérez plusieurs hôtes connectés dans un réseau et regroupez les ressources des hôtes.

| Port
| Configurez le port du serveur vCenter ou ESXi.

| Nom d'utilisateur et mot de passe
| Entrez votre nom d'utilisateur et votre mot de passe de connexion vSphere.
|===

[#_scheduling]
== Planification

Choisissez à quel hyperviseur la machine virtuelle sera planifiée.

Les champs de la section *Planification* devraient se remplir automatiquement avec le centre de données et d'autres options de planification qui vous sont disponibles dans vSphere.

|===
| Champ | Requis | Explication

| Centre de données
| *
| Choisissez le nom ou le chemin du centre de données où la VM sera planifiée.

| Pool de ressources
|
| Nom du pool de ressources pour planifier les VMs. Les pools de ressources peuvent être utilisés pour partitionner les ressources UC et mémoire disponibles d'un hôte autonome ou d'un cluster, et ils peuvent également être imbriqués. Laissez vide pour un ESXi autonome. Si non spécifié, le pool de ressources par défaut est utilisé.

| Zone de stockage des données
| *
| Si vous avez un cluster de magasins de données, vous pouvez activer le champ *Magasin de données*. Cela vous permet de sélectionner un cluster de magasins de données où votre VM sera planifiée. Si le champ n'est pas activé, vous pouvez sélectionner un disque individuel.

| Dossier
|
| Nom d'un dossier dans le centre de données pour créer les VMs. Doit déjà exister. Les dossiers de VM dans ce menu déroulant correspondent directement à vos dossiers de VM dans vSphere. Le nom du dossier doit être précédé de `vm/` dans votre fichier de configuration vSphere.

| Hôte
|
| L'IP du système hôte pour planifier les VMs. Laissez ce champ vide pour un ESXi autonome ou pour un cluster avec DRS (Distributed Resource Scheduler). Si spécifié, le pool du système hôte sera utilisé et le paramètre *Pool de ressources* sera ignoré.

| Délai d'arrêt gracieux
|
| Le temps, en secondes, que Rancher attend avant de supprimer des machines virtuelles sur un cluster. Si défini sur `0`, l'arrêt gracieux est désactivé. N'accepte que des valeurs entières.
|===

[#_instance_options]
== Options d'instance

Dans la section *Options d'instance*, configurez le nombre de vCPUs, la mémoire et la taille du disque pour les VMs créées par ce modèle.

[cols="<,^,<"]
|===
| Paramètre | Requis | Description

| Nombre de processeurs
| *
| Nombre de vUC à attribuer aux VMs.

| Mémoire
| *
| Quantité de mémoire à attribuer aux VMs.

| Disque
| *
| Taille du disque (en Mo) à attacher aux VMs.

| Méthode de création
| *
| La méthode pour configurer un système d'exploitation sur le nœud. Le système d'exploitation peut être installé à partir d'un ISO ou d'un modèle de VM. Selon la méthode de création, vous devrez également spécifier un modèle de VM, une bibliothèque de contenu, une VM existante ou un ISO. Pour plus d'informations sur les méthodes de création, référez-vous à <<_about_vm_creation_methods,À propos des méthodes de création de VM.>>

| Cloud Init
|
| URL d'un fichier `cloud-config.yml` ou URL pour provisionner des VMs. Ce fichier permet une personnalisation supplémentaire du système d'exploitation, comme la configuration réseau, les serveurs DNS ou les daemons système. Le système d'exploitation doit prendre en charge `cloud-init`.

| Réseaux
|
| Nom(s) du réseau auquel attacher la VM.

| Paramètres de configuration utilisés pour guestinfo
|
| Paramètres de configuration supplémentaires pour les VMs. Ceci correspond aux https://kb.vmware.com/s/article/1016098[Paramètres avancés] dans la console vSphere. Des cas d'utilisation typiques incluent la fourniture de paramètres https://rancher.com/docs/os/v1.x/en/installation/cloud/vmware-esxi/#vmware-guestinfo[guestinfo] pour RancherOS ou l'activation des UUID de disque pour les VMs (`disk.EnableUUID=TRUE`).
|===

[#_about_vm_creation_methods]
=== À propos des méthodes de création de VM

Dans le champ *Méthode de création*, configurez la méthode utilisée pour provisionner des VMs dans vSphere. Les options disponibles incluent la création de VMs qui démarrent à partir d'une ISO RancherOS ou la création de VMs par clonage à partir d'une machine virtuelle existante ou d'un https://docs.vmware.com/en/VMware-vSphere/6.5/com.vmware.vsphere.vm_admin.doc/GUID-F7BF0E6B-7C4F-4E46-8BBF-76229AEA7220.html[modèle de VM].

La VM ou le modèle existant peut utiliser tout système d'exploitation Linux moderne configuré pour prendre en charge https://cloudinit.readthedocs.io/en/latest/[cloud-init] en utilisant le https://canonical-cloud-init.readthedocs-hosted.com/en/latest/reference/datasources/nocloud.html[datasource NoCloud].

Choisissez la manière dont la VM sera créée{nbsp}:

* *Déployer à partir d'un modèle{nbsp}: Data Center:* Choisissez un modèle de VM qui existe dans le centre de données que vous avez sélectionné.
* *Déployer à partir d'un modèle{nbsp}: Bibliothèque de contenu :* Tout d'abord, sélectionnez la https://docs.vmware.com/en/VMware-vSphere/6.5/com.vmware.vsphere.vm_admin.doc/GUID-254B2CE8-20A8-43F0-90E8-3F6776C2C896.html[Bibliothèque de contenu] qui contient votre modèle, puis sélectionnez le modèle dans la liste peuplée *Modèles de bibliothèque*.
* *Cloner une machine virtuelle existante :* Dans le champ *Machine virtuelle*, choisissez une VM existante dont la nouvelle VM sera clonée.
* *Installer à partir de l'ISO boot2docker :* Assurez-vous que le champ *URL de l'ISO du système d'exploitation* contient l'URL d'une version ISO VMware pour RancherOS (`rancheros-vmware.iso`). Notez que cette URL doit être accessible depuis les nœuds exécutant votre installation du serveur Rancher.

[#_networks]
== Réseaux

Le modèle de nœud permet désormais de provisionner une VM avec plusieurs réseaux. Dans le champ *Réseaux*, vous pouvez maintenant cliquer sur *Ajouter un réseau* pour ajouter tous les réseaux disponibles dans vSphere.

[#_node_tags_and_custom_attributes]
== Étiquettes de nœud et attributs personnalisés

Les étiquettes vous permettent d'attacher des métadonnées aux objets dans l'inventaire vSphere pour faciliter le tri et la recherche de ces objets.

Pour les étiquettes, toutes vos étiquettes vSphere apparaîtront comme options à sélectionner dans votre modèle de nœud.

Dans les attributs personnalisés, Rancher vous permettra de sélectionner tous les attributs personnalisés que vous avez déjà configurés dans vSphere. Les attributs personnalisés sont des clés et vous pouvez entrer des valeurs pour chacun d'eux.

[NOTE]
====

Les attributs personnalisés sont une fonctionnalité héritée qui sera éventuellement supprimée de vSphere.
====


[#_cloud_init]
== cloud-init

https://cloudinit.readthedocs.io/en/latest/[Cloud-init] vous permet d'initialiser vos nœuds en appliquant une configuration au premier démarrage. Cela peut impliquer des choses telles que la création d'utilisateurs, l'autorisation de clés SSH ou la configuration du réseau.

Pour utiliser l'initialisation cloud-init, créez un fichier de configuration cloud en utilisant une syntaxe YAML valide et collez le contenu du fichier dans le champ *Cloud Init*. Référez-vous à la https://cloudinit.readthedocs.io/en/latest/topics/examples.html[documentation cloud-init.] pour un ensemble d'exemples commentés de directives de configuration cloud prises en charge.

Notez que cloud-init n'est pas pris en charge lors de l'utilisation de la méthode de création ISO.

[#_engine_options]
== Options du moteur

Dans la section *Options du moteur* du modèle de nœud, vous pouvez configurer le daemon de conteneur. Vous souhaiterez peut-être spécifier la version du conteneur ou un miroir de registre d'images de conteneur.

[NOTE]
====
Si vous provisionnez des nœuds Red Hat Enterprise Linux (RHEL) ou CentOS, laissez le champ *URL d'installation de Docker* à la valeur par défaut, ou sélectionnez *aucun*. Cela contournera une vérification de l'installation de Docker, car Docker est déjà installé sur ces types de nœuds.

Si vous définissez *URL d'installation de Docker* à une valeur autre que la valeur par défaut ou *aucun*, vous pourriez voir un message d'erreur tel que le suivant{nbsp}: `Error creating machine: RHEL ssh command error: command: sudo -E yum install -y curl err: exit status 1 output: Updating Subscription Management repositories.`
====
