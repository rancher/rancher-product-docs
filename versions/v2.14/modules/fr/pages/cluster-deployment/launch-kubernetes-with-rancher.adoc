Rancher vous permet de lancer un cluster Kubernetes sur différents types de nœuds, y compris des serveurs sur du matériel sans système d'exploitation, des machines virtuelles sur site et des machines virtuelles d'un fournisseur d'infrastructure. Lors du déploiement de Kubernetes sur ces nœuds, Rancher vous donne la possibilité d'utiliser {url-rke2-introduction}[RKE2] ou {url-k3s}[K3s].

Rancher peut installer Kubernetes sur des nœuds existants, ou il peut provisionner dynamiquement des nœuds dans un fournisseur d'infrastructure et y installer Kubernetes.

Rancher peut également créer des pools de machines. Un avantage d'installer Kubernetes sur des pools de machines hébergés par un fournisseur d'infrastructure est que si un nœud perd la connectivité avec le cluster, Rancher peut automatiquement créer un autre nœud pour rejoindre le cluster afin de garantir que le nombre de machines dans le pool est conforme aux attentes.

[#__rke2_product_name]
== {rke2-product-name}

Rancher v2.6 a introduit le provisionnement pour les clusters {url-rke2-introduction}[RKE2] directement depuis l'interface utilisateur de Rancher. RKE2, également connu sous le nom de RKE Government, est une distribution Kubernetes entièrement conforme axée sur la sécurité et la conformité aux États-Unis. secteur du gouvernement fédéral américain. Dans Rancher v.2.6.4 et les versions antérieures, le provisionnement RKE2 était en aperçu technique.

Notez que dans Rancher v2.6.5, le provisionnement RKE2 est devenu GA.

[#_requirements]
=== Configuration requise

Si vous utilisez RKE2 pour configurer un cluster, vos nœuds doivent répondre aux {url-rke2-requirements}[exigences] pour les nœuds dans les clusters utilisateurs en aval.

[#_launching_kubernetes_on_new_nodes_in_an_infrastructure_provider]
=== Lancement de Kubernetes sur de nouveaux nœuds dans un fournisseur d'infrastructure

Le provisionnement RKE2 est construit sur un nouveau cadre de provisionnement qui tire parti du projet en amont https://github.com/kubernetes-sigs/cluster-api[Cluster API (CAPI)]. Avec ce nouveau cadre de provisionnement, vous pouvez{nbsp}:

* Provisionner des clusters RKE2 sur tout fournisseur pour lequel Rancher dispose d'un pilote de nœud.
* Configurer entièrement les clusters RKE2 au sein de Rancher.
* Choisir les options CNI Calico, Cilium et Multus en plus de Canal.

Lorsque vous apportez des modifications à la configuration de votre cluster dans RKE2, cela peut entraîner le reprovisionnement des nœuds. Cela est contrôlé par les contrôleurs CAPI et non par Rancher lui-même. Notez que pour les nœuds etcd, le même comportement ne s'applique pas.

Un exemple de changement de configuration qui peut provoquer le comportement décrit est lorsque l'on modifie un cluster et que l'on active le drainage avant la suppression. Dans ce scénario, les nœuds du plan de contrôle existants et les nœuds de travail sont supprimés et de nouveaux nœuds sont créés.

Le provisionnement RKE2 inclut également l'installation de RKE2 sur des clusters avec des nœuds Windows.

Les fonctionnalités Windows pour RKE2 incluent{nbsp}:

* Windows prend en charge le pilote de nœud vSphere.
* Calico et Flannel CNI pour les clusters personnalisés RKE2 sous Windows.
* Isolation de réseau de projet (PNI) pour Calico.
* Conteneurs Windows avec RKE2 alimenté par containerd.
* Provisionnement de clusters Windows RKE2 via Terraform.
* Provisionnement de clusters personnalisés Windows RKE2 directement depuis l'interface utilisateur de Rancher.

Le support Windows pour les clusters personnalisés RKE2 nécessite de choisir Calico comme CNI.

[#_launching_kubernetes_on_existing_custom_nodes]
=== Lancement de Kubernetes sur des nœuds personnalisés existants.

Le provisionnement RKE2 permet également d'installer des clusters personnalisés sur des machines virtuelles ou des nœuds de matériel sans système d'exploitation précédemment provisionnés.

Si vous souhaitez réutiliser un nœud d'un cluster personnalisé précédent, {xref-clean-cluster-nodes}#_cleaning_up_nodes[nettoyez le nœud] avant de l'utiliser à nouveau dans un cluster. Si vous réutilisez un nœud qui n'a pas été nettoyé, le provisionnement du cluster peut échouer.

[#_programmatically_creating_rke2_product_name_clusters]
=== Création programmatique de {rke2-product-name} clusters.

La manière la plus courante de déployer des clusters RKE2 de manière programmatique via Rancher est d'utiliser le https://registry.terraform.io/providers/rancher/rancher2/latest/docs/resources/cluster_v2[fournisseur Terraform Rancher2].
