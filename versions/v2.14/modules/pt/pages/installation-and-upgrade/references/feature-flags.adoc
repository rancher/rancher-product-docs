Com os sinalizadores de recurso, você pode experimentar recursos opcionais ou experimentais e habilitar recursos legados que estão sendo descontinuados.

Para saber mais sobre os valores dos recursos e como habilitá-los, consulte {xref-experimental-features}[Habilitando Recursos Experimentais].

[NOTE]
====
Alguns sinalizadores de recurso requerem uma reinicialização do contêiner do Rancher. Sinalizadores de recurso que requerem reinicialização estão marcados na interface do usuário do Rancher.
====


A seguir, está uma lista de sinalizadores de recurso disponíveis no Rancher. Se você atualizou de uma versão anterior do Rancher, pode ver sinalizadores adicionais na interface do usuário do Rancher, como `proxy` ou `dashboard` (ambos xref:[descontinuados]):

* `aggregated-roletemplates`: Use a arquitetura de agregação de funções de cluster para RoleTemplates, ProjectRoleTemplateBindings e ClusterRoleTemplateBindings. Veja {xref-role-template-aggregation}[Agregação de RoleTemplate] para mais informações.
* `clean-stale-secrets`: Remove segredos obsoletos do namespace `cattle-impersonation-system`. Isso limpa lentamente segredos antigos que não estão mais sendo usados pelo sistema de impersonação.
* `continuous-delivery`: Permite que o Fleet GitOps seja desativado separadamente do Fleet. Veja {xref-continuous-delivery}[Entrega Contínua] para mais informações.
* `fleet`: O framework de provisionamento do Rancher na versão v2.6 e posteriores requer o Fleet. O sinalizador de recurso será habilitado automaticamente quando você atualizar, mesmo que você tenha desativado esse sinalizador em uma versão anterior do Rancher. Veja {xref-fleet}[Entrega Contínua com Fleet] para mais informações.
* `harvester`: Gerencia o acesso à página de Gerenciamento de Virtualização, onde os usuários podem navegar diretamente para clusters Harvester e acessar a interface do Harvester. Veja {xref-harvester-overview}[{harvester-product-name} Visão Geral da Integração] para mais informações.
* `imperative-api-extension`: Habilita o https://kubernetes.io/docs/concepts/extend-kubernetes/api-extension/apiserver-aggregation/[servidor da API de extensão] do Rancher para registrar novas APIs no Kubernetes. Esse sinalizador de recurso está habilitado por padrão. Veja a página xref:api/extension-apiserver.adoc[Servidor da API de Extensão] para mais informações.
* `istio-virtual-service-ui`: Habilita uma {xref-istio-traffic-management}[interface visual] para criar, ler, atualizar e excluir serviços virtuais e regras de destino do Istio, que são recursos de gerenciamento de tráfego do Istio.
* `legacy`: Habilita um conjunto de recursos da versão 2.5.x e anteriores, que estão sendo gradualmente descontinuados em favor de implementações mais novas. Esses são uma mistura de recursos descontinuados, bem como recursos que eventualmente estarão disponíveis em versões mais novas. Esse sinalizador de recurso está desabilitado por padrão em novas instalações do Rancher. Se você está atualizando de uma versão anterior do Rancher, esse sinalizador de recurso está habilitado.
* `managed-system-upgrade-controller`: Habilita a instalação do app system-upgrade-controller em clusters downstream importados RKE2/K3s, bem como no cluster local, se este for um cluster RKE2/K3s.

[IMPORTANT]
====
Este `managed-system-upgrade-controller` sinalizador de recurso é destinado a **uso interno apenas** e não possui um CR de recurso associado. Use com cautela.

Para controlar se o Rancher deve gerenciar a versão do Kubernetes de clusters RKE2/K3s importados, é recomendável usar a {xref-register-clusters}#_configurando_o_gerenciamento_de_versão_para_clusters_rke2_e_k3s[funcionalidade_de_gerenciamento_de_versão_de_cluster_importado] que está disponível no Rancher v2.11.0 ou mais recente.
====

[WARNING]
====
Se o `managed-system-upgrade-controller` sinalizador de recurso foi **desativado** no Rancher v2.10.x, e qualquer cluster RKE2/K3s importado foi atualizado **fora do Rancher**, siga os passos abaixo para evitar a instalação inesperada do app system-upgrade-controller e garantir que a {xref-register-clusters}#_configurando_o_gerenciamento_de_versão_para_clusters_rke2_e_k3s[funcionalidade_de_gerenciamento_de_versão_de_cluster_importado] funcione corretamente:

. Atualize o Rancher para v2.11.0 ou mais recente, certificando-se de **manter** o sinalizador de recurso representado por `managed-system-upgrade-controller=false` nos valores do Helm, se ele foi definido durante a instalação do v2.10.x.
. Após o Rancher estar totalmente em funcionamento, desative a `imported-cluster-version-management` configuração. Você pode fazer isso através da interface do Rancher clicando em **☰ > Configurações Globais > Configurações > gerenciamento-de-versão-de-cluster-importado**, ou editando o `Setting.management.cattle.io/v3` recurso personalizado correspondente via kubectl.
. Realize uma segunda atualização do Helm, desta vez omitindo o sinalizador de recurso representado por `managed-system-upgrade-controller=false`.

Agora, o gerenciamento de versão do cluster importado está desativado por padrão, e o Rancher não instala mais automaticamente o aplicativo system-upgrade-controller em clusters importados.

Você pode habilitar essa funcionalidade por cluster. Para mais informações, consulte a {xref-register-clusters}#_configurando_o_gerenciamento_de_versão_para_clusters_rke2_e_k3s[documentação].
====

* `multi-cluster-management`: Permite o provisionamento e gerenciamento de clusters Kubernetes multi-cluster. Esse sinalizador de recurso só pode ser definido no momento da instalação. Não pode ser habilitada ou desabilitada posteriormente.
* `rke2`: Habilita o provisionamento de clusters RKE2. Esse sinalizador de recurso está habilitado por padrão.
* `token-hashing`: Habilita o hashing de tokens. Uma vez habilitado, os tokens existentes serão convertidos em hash, e todos os novos tokens serão automaticamente processados com o algoritmo SHA256. Uma vez que um token é convertido em hash, não é possível reverter o processo. Esse sinalizador de recurso não pode ser desativado após ser habilitado. Veja xref:api/api-tokens.adoc#_token_hashing[Tokens de API] para mais informações.
* `uiextension`: Habilita extensões da interface do usuário. Esse sinalizador de recurso está habilitado por padrão. Habilitar ou desabilitar o sinalizador força o pod do Rancher a reiniciar. A primeira vez que este sinalizador é definido como `Active`, ele cria um CRD e habilita os controladores e pontos finais necessários para o recurso funcionar. Se definido como `Disabled`, ele desabilita os controladores e pontos finais mencionados anteriormente. Definir `uiextension` como `Disabled` não tem efeito sobre o CRD --; não cria um CRD se ele ainda não existir, nem exclui o CRD se ele já existir.* `unsupported-storage-drivers`: Habilita tipos para provedores de armazenamento e provisionadores que não estão habilitados por padrão. Veja {xref-unsupported-storage-drivers}[Permitir Drivers de Armazenamento Não Suportados] para mais informações.
* `ui-sql-cache`: Habilita um cache baseado em SQLite para tabelas da interface do usuário e paginação do lado do servidor. Veja {xref-ui-server-side-pagination}[Paginação do Lado do Servidor da UI] para mais informações.
