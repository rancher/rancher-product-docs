Você pode visualizar e manipular todos os recursos personalizados e CRDs em um cluster Kubernetes pela interface do Rancher.

[#_workloads]
== Cargas de trabalho

Implante aplicativos em seus nós de cluster usando {xref-workloads-and-pods}[workloads], que são objetos que contêm pods que executam seus aplicativos, juntamente com metadados que definem regras para o comportamento da implantação. Os workloads podem ser implantados no escopo de todo o cluster ou dentro de um namespace.

Ao implantar um workload, você pode implantar a partir de qualquer imagem. Há uma variedade de {xref-workloads-and-pods}#_workload_types[_workload_types[tipos de workload] para escolher, que determinam como seu aplicativo deve ser executado.

Após a implantação de um workload, você pode continuar trabalhando com ele. Você pode:

* {xref-upgrade-workloads}[Fazer upgrade] o workload para uma versão mais nova do aplicativo que está sendo executado.
* {xref-roll-back-workloads}[Reverter] um workload para uma versão anterior, se ocorrer um problema durante o processo de fazer upgrade.
* {xref-add-sidecar}[Adicionar um sidecar], que é um workload que suporta um workload principal.

[#_load_balancing_and_ingress]
== Balanceamento de Carga e Ingress

[#_load_balancers]
=== Balanceadores de Carga

Após lançar um aplicativo, ele está disponível apenas dentro do cluster. Não pode ser acessado externamente.

Se você quiser que seus aplicativos sejam acessíveis externamente, deve adicionar um balanceador de carga ao seu cluster. Balanceadores de carga criam um gateway para conexões externas acessarem seu cluster, desde que o usuário conheça o endereço IP do balanceador de carga e o número da porta do aplicativo.

O Rancher suporta dois tipos de balanceadores de carga:

* {xref-l4-l7-lb}#_layer_4_load_balancer[Balanceadores de Carga de Camada 4]
* {xref-l4-l7-lb}#_layer_7_load_balancer[Balanceadores de Carga de Camada 7]

Para mais informações, veja {xref-l4-l7-lb}[balanceadores de carga].

[#_ingress]
==== Ingress

Os Balanceadores de Carga podem lidar com apenas um endereço IP por serviço, o que significa que se você executar vários serviços em seu cluster, deve ter um balanceador de carga para cada serviço. Executar múltiplos balanceadores de carga pode ser caro. Você pode contornar esse problema usando um ingress.

Ingress é um conjunto de regras que atua como um balanceador de carga. O ingress funciona em conjunto com um ou mais controladores de ingress para direcionar dinamicamente as solicitações de serviço. Quando o ingress recebe uma solicitação, o(s) controlador(es) de ingress em seu cluster programam o balanceador de carga para direcionar a solicitação para o serviço correto com base nos subdomínios de serviço ou nas regras de caminho que você configurou.

Para mais informações, consulte {xref-add-ingresses}[Ingress].

Ao usar ingresses em um projeto, você pode programar o nome do host do ingress para um DNS externo configurando uma entrada de DNS Global.

[#_service_discovery]
== Descoberta de Serviços

Depois de expor seu cluster a solicitações externas usando um balanceador de carga e/ou ingress, ele só está disponível por endereço IP. Para criar um nome de host resolvível, você deve criar um registro de serviço, que é um registro que mapeia um endereço IP, nome de host externo, álias de registro DNS, carga(s) de trabalho ou pods rotulados a um nome de host específico.

Para mais informações, consulte {xref-create-services}[Descoberta de Serviços].

[#_applications]
== Aplicativos

Além de lançar componentes individuais de uma aplicação, você pode usar o catálogo do Rancher para começar a lançar aplicações, que são charts do Helm.

[#_kubernetes_resources]
== Kubernetes Resources

Dentro do contexto de um projeto ou namespace do Rancher, _recursos_ são arquivos e dados que suportam a operação de seus pods. Dentro do Rancher, certificados, registros e segredos são todos considerados recursos. No entanto, o Kubernetes classifica recursos como diferentes tipos de https://kubernetes.io/docs/concepts/configuration/secret/[segredos]. Portanto, dentro de um único projeto ou namespace, recursos individuais devem ter nomes únicos para evitar conflitos. Embora os recursos sejam usados principalmente para transportar informações sensíveis, eles também têm outros usos.

Os recursos incluem:

* {xref-http-encrypt}[Certificados]: Arquivos usados para criptografar/descriptografar dados que entram ou saem do cluster.
* {xref-configmaps}[ConfigMaps]: Arquivos que armazenam informações gerais de configuração, como um grupo de arquivos de configuração.
* {xref-secrets}[Segredos]: Arquivos que armazenam dados sensíveis como senhas, tokens ou chaves.
* {xref-k8s-docker-registries}[Registros]: Arquivos que contêm credenciais usadas para autenticar em registros privados.
