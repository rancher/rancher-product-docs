Ao implantar um aplicativo que precisa reter dados, você precisará criar armazenamento persistente. O armazenamento persistente permite que você armazene dados do aplicativo externamente ao pod que está executando seu aplicativo. Essa prática de armazenamento permite que você mantenha os dados do aplicativo, mesmo que o pod do aplicativo falhe.

Os documentos nesta seção assumem que você entende os conceitos do Kubernetes de volumes persistentes, reivindicações de volumes persistentes e classes de armazenamento. Para mais informações, consulte a seção sobre {xref-about-persistent-storage}[como o armazenamento funciona.]

[#_prerequisites]
== Pré-requisitos

Para configurar o armazenamento persistente, o `Manage Volumes` {xref-cluster-project-roles}#_project_role_reference[role] é necessário.

Se você estiver provisionando armazenamento para um cluster hospedado na nuvem, o armazenamento e os hosts do cluster devem ter o mesmo provedor de nuvem.

Para provisionar novo armazenamento com o Rancher, o provedor de nuvem deve estar habilitado. Para detalhes sobre como habilitar provedores de nuvem, consulte {xref-set-up-cloud-providers}[esta página.]

Para anexar armazenamento persistente existente a um cluster, o provedor de nuvem não precisa estar habilitado.

[#_setting_up_existing_storage]
== Configurando Armazenamento Existente

O fluxo de trabalho geral para configurar armazenamento existente é o seguinte:

. Configure seu armazenamento persistente. Isso pode ser armazenamento em um provedor de infraestrutura, ou pode ser seu próprio armazenamento.
. Adicione um volume persistente (PV) que faça referência ao armazenamento persistente.
. Adicione uma reivindicação de volume persistente (PVC) que faça referência ao PV.
. Montar o PVC como um volume em sua carga de trabalho.

Para detalhes e pré-requisitos, consulte {xref-existing-storage-setup}[esta página.]

[#_dynamically_provisioning_new_storage_in_rancher]
== Provisionamento Dinâmico de Novo Armazenamento no Rancher

O fluxo de trabalho geral para provisionar novo armazenamento é o seguinte:

. Adicione uma StorageClass e configure-a para usar seu provedor de armazenamento. A StorageClass pode se referir ao armazenamento em um provedor de infraestrutura ou pode se referir ao seu próprio armazenamento.
. Adicione uma reivindicação de volume persistente (PVC) que se refere à classe de armazenamento.
. Monte o PVC como um volume para sua carga de trabalho.

Para detalhes e pré-requisitos, consulte {xref-dynamically-provision-new-storage}[esta página.]

[#__longhorn_product_name_storage]
== {longhorn-product-name} Armazenamento

https://documentation.suse.com/cloudnative/storage/{longhorn-docs-version}/en/longhorn-documentation.html[{longhorn-product-name}] é um sistema de armazenamento em blocos distribuído, leve, confiável e fácil de usar para Kubernetes.

Longhorn é um software de código aberto. Desenvolvido originalmente pela Rancher Labs, agora está sendo desenvolvido como um projeto sandbox da Cloud Native Computing Foundation. Pode ser instalado em qualquer cluster Kubernetes com Helm, com kubectl ou com a interface do usuário do Rancher.

Se você tiver um pool de armazenamento em blocos, o Longhorn pode ajudá-lo a fornecer armazenamento persistente para seu cluster Kubernetes sem depender de provedores de nuvem. Para mais informações sobre os recursos do Longhorn, consulte a https://documentation.suse.com/cloudnative/storage/{longhorn-docs-version}/en/introduction/introduction.html[documentação.]

O Rancher v2.5 simplificou o processo de instalação do Longhorn em um cluster gerenciado pelo Rancher. Para mais informações, veja {xref-longhorn}[Armazenamento Nativo de Nuvem com Longhorn].

[#_glusterfs_volumes]
== GlusterFS Volumes

Em clusters que armazenam dados em volumes GlusterFS, você pode enfrentar um problema em que os pods falham ao montar volumes após reiniciar o `kubelet`. Para detalhes sobre como evitar que isso aconteça, consulte {xref-glusterfs-storage}[esta página.]

[#_iscsi_volumes]
== Volumes iSCSI

Em {xref-launch-k8s-with-rancher}[Clusters Kubernetes Lançados pelo Rancher] que armazenam dados em volumes iSCSI, você pode enfrentar um problema onde os kubelets falham ao se conectar automaticamente com os volumes iSCSI. Para detalhes sobre como resolver esse problema, consulte {xref-iscsi-storage}[esta página.]

[#_hostpath_volumes]
== Volumes hostPath

Tanto o K3s quanto o RKE2 suportam a montagem de volumes hostPath usando o https://github.com/rancher/local-path-provisioner[Rancher Local Path Provisioner]. Para informações de configuração, dependendo da sua distribuição, consulte {url-k3s-hostpath}[K3s - Volumes e Armazenamento] ou {url-rke2-hostpath}[RKE2 - Opções Avançadas e Configuração].

[#_migrating_vmware_vsphere_cloud_provider_from_in_tree_to_out_of_tree]
== Migrando o Provedor de Nuvem VMware vSphere de In-tree para Out-of-tree

O Kubernetes está se afastando da manutenção de provedores de nuvem in-tree. O vSphere possui um provedor de nuvem out-of-tree que pode ser utilizado instalando o provedor de nuvem vSphere e os plugins de armazenamento em nuvem.

Para instruções sobre como migrar do provedor de nuvem vSphere in-tree para out-of-tree e gerenciar as VMs existentes após a migração, consulte {xref-cloud-provider-vsphere-out-of-tree}[esta página.]
