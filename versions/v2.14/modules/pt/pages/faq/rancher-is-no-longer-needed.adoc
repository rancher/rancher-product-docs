Esta página tem como objetivo responder perguntas sobre o que acontece se você não quiser mais o Rancher, se não quiser que um cluster seja gerenciado pelo Rancher ou se o servidor Rancher for excluído.

[#_if_the_rancher_server_is_deleted_what_happens_to_the_workloads_in_my_downstream_clusters]
== Se o servidor Rancher for excluído, o que acontece com as cargas de trabalho nos meus clusters downstream?

Se o Rancher for excluído ou se tornar irrecuperável, todas as cargas de trabalho nos clusters Kubernetes downstream gerenciados pelo Rancher continuarão a funcionar normalmente.

[#_if_the_rancher_server_is_deleted_how_do_i_access_my_downstream_clusters]
== Se o servidor Rancher for excluído, como faço para acessar meus clusters downstream?

A capacidade de acessar um cluster downstream sem o Rancher depende do tipo de cluster e da forma como o cluster foi criado. Resumindo:

* *Clusters registrados/importados:* O cluster não será afetado e você pode acessá-lo usando os mesmos métodos que utilizou antes de registrar o cluster no Rancher.
ifeval::["{build-type}" != "srfa"] 
* *Clusters Kubernetes hospedados:* Se você criou o cluster em um provedor de Kubernetes hospedado na nuvem, como EKS, GKE ou AKS, pode continuar a gerenciar o cluster usando as credenciais de nuvem do seu provedor.
endif::[]
ifeval::["{build-type}" == "srfa"]
* *Clusters Kubernetes hospedados:* Se você criou o cluster em um provedor de Kubernetes hospedado na nuvem, como EKS, pode continuar a gerenciar o cluster usando as credenciais de nuvem do seu provedor.
endif::[]
* *Clusters provisionados pelo Rancher:* Para acessar um {xref-launch-k8s-with-rancher}[cluster RKE2/K3s,] o cluster deve ter o {xref-arch-com-ds-cluster}#_4_endpoint_de_cluster_autorizado[endpoint de cluster autorizado] habilitado, e você deve ter baixado o arquivo kubeconfig do cluster na interface do usuário do Rancher. (O endpoint de cluster autorizado é habilitado por padrão para clusters RKE.) Com este endpoint, você pode acessar seu cluster com o kubectl diretamente, em vez de se comunicar através do {xref-arch-com-ds-cluster}#_1_proxy_de_autenticação[proxy de autenticação] do servidor Rancher. Para instruções sobre como configurar o kubectl para usar o endpoint de cluster autorizado, consulte a seção sobre como acessar clusters diretamente com {xref-access-cluster-kubectl}#_autenticando_diretamente_com_um_cluster_downstream[kubectl e o arquivo kubeconfig]. Esses clusters usarão um instantâneo da autenticação conforme foi configurado quando o Rancher foi removido.

[#_what_if_i_dont_want_rancher_anymore]
== E se eu não quiser mais o Rancher?

[NOTE]
====

As xref:[Ferramentas do Sistema] anteriormente recomendadas foram descontinuadas desde junho de 2022.
====


Se você {xref-install-rancher}[instalou o Rancher em um cluster Kubernetes,] remova o Rancher usando a ferramenta https://github.com/rancher/rancher-cleanup[Rancher Cleanup].

Desinstalar o Rancher em modo de alta disponibilidade (HA) também removerá todos os pods `helm-operation-*` e os seguintes apps:

* fleet
* fleet-agent
* rancher-operator
* rancher-webhook

Recursos personalizados (CRDs) e namespaces personalizados ainda precisarão ser removidos manualmente.

Se você instalou o Rancher com Docker, pode desinstalar o Rancher removendo o único contêiner Docker em que ele é executado.

Clusters importados não serão afetados pela remoção do Rancher. Para outros tipos de clusters, consulte a seção sobre <<_if_the_rancher_server_is_deleted_how_do_i_access_my_downstream_clusters,acesso a clusters downstream quando o Rancher for removido.>>

[#_what_if_i_dont_want_my_registered_cluster_managed_by_rancher]
== E se eu não quiser que meu cluster registrado seja gerenciado pelo Rancher?

Se um cluster registrado for excluído da interface do Rancher, o cluster será desanexado do Rancher, permanecendo intacto e acessível pelos mesmos métodos que foram usados para acessá-lo antes de ser registrado no Rancher.

Para desanexar o cluster,

. No canto superior esquerdo, clique em *☰ > Gerenciamento de Cluster*.
. Vá para o cluster registrado que deve ser desanexado do Rancher e clique em *⋮ > Excluir*.
. Clique em *Excluir*.

*Resultado:* O cluster registrado é desanexado do Rancher e funciona normalmente fora do Rancher.

[#_what_if_i_dont_want_my_hosted_kubernetes_cluster_managed_by_rancher]
== E se eu não quiser que meu cluster Kubernetes hospedado seja gerenciado pelo Rancher?

Neste momento, não há funcionalidade para desanexar esses clusters do Rancher. Neste contexto, "desanexar" é definido como a capacidade de remover componentes do Rancher do cluster e gerenciar o acesso ao cluster de forma independente do Rancher.

A capacidade de gerenciar esses clusters sem o Rancher está sendo acompanhada neste https://github.com/rancher/rancher/issues/25234[problema.]

Para informações sobre como acessar clusters se o servidor Rancher for excluído, consulte <<_if_the_rancher_server_is_deleted_how_do_i_access_my_downstream_clusters,esta seção.>>
