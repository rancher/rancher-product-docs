.ソリューションの概要

[#_1_vm_considerations]
== 1.VMの考慮事項

[#_leverage_vm_templates_to_construct_the_environment]
=== VMテンプレートを活用して環境を構築する

環境全体で展開される仮想マシンの一貫性を促進するために、VMテンプレートの形でゴールデンイメージの使用を検討してください。これを達成するためにPackerを使用することができ、より多くのカスタマイズオプションを追加します。

[#_leverage_drs_anti_affinity_rules_where_possible_to_separate_downstream_cluster_nodes_across_esxi_hosts]
=== DRSアンチアフィニティルール（可能な場合）を活用して、ダウンストリームクラスターのノードをESXiホスト間で分離します。

これにより、ノードVMが複数のESXiホストに分散され、ホストレベルでの単一障害点を防ぐことができます。

[#_leverage_drs_anti_affinity_rules_where_possible_to_separate_downstream_cluster_nodes_across_datastores]
=== DRSアンチアフィニティルール（可能な場合）を活用して、ダウンストリームクラスターのノードをデータストア間で分離します。

これにより、ノードVMが複数のデータストアに分散され、データストアレベルでの単一障害点を防ぐことができます。

[#_configure_vms_as_appropriate_for_kubernetes]
=== Kubernetesに適したVMを構成する

ノードを展開する際には、スワップを無効にし、クラスター内のすべてのマシン間で完全なネットワーク接続があることを再確認し、各ノードに対してユニークなホスト名、MACアドレス、およびproduct_uuidを使用するなど、K8sおよびetcdのベストプラクティスに従うことが重要です。

[#_2_network_considerations]
== 2.ネットワークの考慮事項

[#_leverage_low_latency_high_bandwidth_connectivity_between_etcd_nodes]
=== ETCDノード間の低遅延、高帯域幅の接続を活用する

可能な限り、単一のデータセンター内にETCDメンバーをデプロイすることで、遅延オーバーヘッドを回避し、ネットワーク分割の可能性を減らします。ほとんどのセットアップでは、1Gbの接続で十分です。大規模なクラスターの場合、10Gbの接続によりバックアップからの復元にかかる時間を短縮できます。

[#_consistent_ip_addressing_for_vms]
=== VMの一貫したIPアドレス設定

使用する各ノードには静的IPが設定されている必要があります。DHCPの場合、各ノードにはDHCP予約が必要で、ノードが同じIPを割り当てられることを確認します。

[#_3_storage_considerations]
== 3.ストレージの考慮事項

[#_leverage_ssd_drives_for_etcd_nodes]
=== ETCDノードにSSDドライブを活用する

ETCDは書き込み遅延に非常に敏感です。したがって、可能な限りSSDを活用してください。

[#_4_backups_and_disaster_recovery]
== 4.バックアップと災害復旧

[#_perform_regular_downstream_cluster_backups]
=== 定期的なダウンストリームクラスターのバックアップを実施してください。

Kubernetesは、すべてのデータ（設定、状態、メタデータ）を保存するためにetcdを使用します。災害復旧の際にこれをバックアップすることは重要です。

[#_back_up_downstream_node_vms]
=== ダウンストリームノードVMをバックアップしてください。

標準のVMバックアップポリシーにRancherのダウンストリームノードVMを組み込んでください。
