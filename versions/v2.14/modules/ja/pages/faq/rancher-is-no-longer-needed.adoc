このページは、Rancherをもう必要としない場合、Rancherによって管理されるクラスタをもう必要としない場合、またはRancherサーバーが削除された場合に何が起こるかについての質問に答えることを目的としています。

[#_if_the_rancher_server_is_deleted_what_happens_to_the_workloads_in_my_downstream_clusters]
== Rancherサーバーが削除された場合、私のダウンストリームクラスタのワークロードには何が起こりますか？

Rancherが削除されるか、復元不可能になった場合、Rancherによって管理されているダウンストリームのKubernetesクラスタ内のすべてのワークロードは、通常通り機能し続けます。

[#_if_the_rancher_server_is_deleted_how_do_i_access_my_downstream_clusters]
== Rancherサーバーが削除された場合、どのようにして私のダウンストリームクラスタにアクセスしますか？

Rancherなしでダウンストリームクラスタにアクセスする能力は、クラスタの種類とクラスタが作成された方法に依存します。これまでの内容を次に要約します。

* *登録済み/インポート済みクラスタ:*クラスタは影響を受けず、Rancherにクラスタが登録される前と同じ方法でクラスタにアクセスできます。
ifeval::["{build-type}" != "srfa"] 
* *ホスト型Kubernetesクラスタ:*EKS、GKE、またはAKSなどのクラウドホスト型Kubernetesプロバイダーでクラスタを作成した場合、プロバイダーのクラウド認証情報を使用してクラスタを引き続き管理できます。
endif::[]
ifeval::["{build-type}" == "srfa"]
* *ホスト型Kubernetesクラスタ:*EKSなどのクラウドホスト型Kubernetesプロバイダーでクラスタを作成した場合、プロバイダーのクラウド認証情報を使用してクラスタを引き続き管理できます。
endif::[]
* *Rancherプロビジョニングされたクラスタ:*{xref-launch-k8s-with-rancher}[RKE2/K3sクラスタ]にアクセスするには、クラスタに{xref-arch-com-ds-cluster}#_4_authorized_cluster_endpoint[認可されたクラスタエンドポイント]が有効になっている必要があり、Rancher UIからクラスタのkubeconfigファイルをすでにダウンロードしている必要があります。(認可されたクラスタエンドポイントは、RKEクラスタの場合、デフォルトで有効になっています。)このエンドポイントを使用すると、Rancherサーバーの{xref-arch-com-ds-cluster}#_1_the_authentication_proxy[認証プロキシ]を介さずに、kubectlを使用して直接クラスタにアクセスできます。認可されたクラスタエンドポイントを使用するためのkubectlの設定方法については、{xref-access-cluster-kubectl}#_authenticating_directly_with_a_downstream_cluster[kubectlとkubeconfigファイルを使用してクラスタに直接アクセスする]のセクションを参照してください。これらのクラスタは、Rancherが削除されたときに構成されていた認証のスナップショットを使用します。

[#_what_if_i_dont_want_rancher_anymore]
== Rancherをもう必要としない場合はどうなりますか？

[NOTE]
====

以前推奨されていたxref:[システムツール]は、2022年6月以降廃止となりました。
====


{xref-install-rancher}[KubernetesクラスタにRancherをインストールした場合、] https://github.com/rancher/rancher-cleanup[Rancher Cleanup]ツールを使用してRancherを削除します。

高可用性（HA）モードでRancherをアンインストールすると、すべての`helm-operation-*`ポッドと次のアプリが削除されます：

* fleet
* fleet-agent
* rancher-operator
* rancher-webhook

カスタムリソース（CRD）およびカスタムネームスペースは、手動で削除する必要があります。

DockerでRancherをインストールした場合、実行中の単一のDockerコンテナを削除することでRancherをアンインストールできます。

インポートされたクラスタは、Rancherが削除されても影響を受けません。他のタイプのクラスタについては、<<_if_the_rancher_server_is_deleted_how_do_i_access_my_downstream_clusters,Rancherが削除されたときのダウンストリームクラスタへのアクセス>>のセクションを参照してください。

[#_what_if_i_dont_want_my_registered_cluster_managed_by_rancher]
== 登録されたクラスタをRancherに管理されたくない場合はどうすればよいですか？

登録されたクラスタがRancherのUIから削除されると、そのクラスタはRancherから切り離され、登録前にアクセスに使用されていた方法でそのまま保持され、アクセス可能です。

クラスタを切り離すには、

. 左上隅で、*☰ > クラスター管理*をクリックします。
. Rancherから切り離すべき登録されたクラスタに移動し、*⋮ > 削除*をクリックします。
. *削除*をクリックします。

*結果:*登録されたクラスタはRancherから切り離され、Rancherの外で正常に機能します。

[#_what_if_i_dont_want_my_hosted_kubernetes_cluster_managed_by_rancher]
== ホストされたKubernetesクラスタをRancherに管理されたくない場合はどうすればよいですか？

現時点では、これらのクラスタをRancherから切り離す機能はありません。この文脈では、「切り離す」とは、クラスタからRancherコンポーネントを削除し、Rancherとは独立してクラスタへのアクセスを管理する能力を指します。

Rancherなしでこれらのクラスターを管理する能力は、この https://github.com/rancher/rancher/issues/25234[問題]で追跡されています。

Rancherサーバーが削除された場合にクラスターにアクセスする方法についての情報は、<<_if_the_rancher_server_is_deleted_how_do_i_access_my_downstream_clusters,このセクション>>を参照してください。
