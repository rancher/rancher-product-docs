ダウンストリームのKubernetesクラスターを管理するRancherインスタンスに推奨されるデプロイメント戦略は2つあります。それぞれに利点と欠点があります。どちらがあなたのユースケースに最適かについて詳しくお読みください。

[#_hub_spoke_strategy]
== ハブ＆スポーク戦略

'''

このデプロイメントシナリオでは、世界中のKubernetesクラスターを管理する単一のRancherインスタンスがあります。Rancherインスタンスは高可用性のKubernetesクラスター上で実行され、レイテンシの影響を受けることがあります。

[#_pros]
=== 利点

* すべての地域と環境を表示するための単一のコントロールプレーンインターフェース。
* KubernetesはRancherなしで動作することができ、Rancherインスタンスへの接続が失われても耐えることができます。

[#_cons]
=== 欠点

* 低レイテンシの影響を受けます。
* Rancherがダウンすると、新しいサービスのグローバルプロビジョニングは復旧するまで利用できません。ただし、各Kubernetesクラスターは個別に管理を続けることができます。

[#_regional_strategy]
== 地域戦略

'''

地域デプロイメントモデルでは、RancherインスタンスがダウンストリームのKubernetesクラスターに近接して展開されます。

[#_pros_1]
=== 利点

* 他の地域のRancherインスタンスがダウンしても、地域内のRancher機能は稼働し続けます。
* Rancherとダウンストリームクラスター間の低レイテンシが大幅に削減され、Rancherの機能のパフォーマンスが向上します。
* Rancherのアップグレードは地域ごとに独立して行うことができます。

[#_cons_1]
=== 欠点

* 複数のRancherのインストールを管理するためのオーバーヘッド。
* 異なる地域のKubernetesクラスターの可視性を確保するには、複数のインターフェースが必要です。
* Rancherでマルチクラスターアプリを展開するには、各Rancherサーバーに対してその処理を繰り返す必要があります。
