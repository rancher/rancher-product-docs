对于管理下游 Kubernetes 集群的 Rancher 实例，推荐两种部署策略。每种策略都有其优缺点。阅读更多关于哪种策略最适合您的用例的信息。

[#_hub_spoke_strategy]
== 中心辐射策略

'''

在这种部署场景中，单个 Rancher 实例管理全球的 Kubernetes 集群。Rancher 实例将在高可用性 Kubernetes 集群上运行，并且会受到延迟影响。

[#_pros]
=== 优点

* 单一控制平面界面可查看所有区域和环境。
* Kubernetes 不需要 Rancher 来操作，并且可以容忍与 Rancher 实例的连接丢失。

[#_cons]
=== 缺点

* 受网络延迟的影响。
* 如果 Rancher 出现故障，全球新服务的配置将无法进行，直到其恢复。然而，每个 Kubernetes 集群仍然可以单独管理。

[#_regional_strategy]
== 区域策略

'''

在区域部署模型中，Rancher 实例被部署在靠近下游 Kubernetes 集群的位置。

[#_pros_1]
=== 优点

* 如果另一个区域的 Rancher 实例出现故障，区域内的 Rancher 功能仍然可以正常运行。
* Rancher 与下游集群之间的网络延迟大大减少，提高了 Rancher 中功能的性能。
* Rancher 的升级可以在每个区域独立进行。

[#_cons_1]
=== 缺点

* 管理多个 Rancher 安装的开销。
* 在不同区域的 Kubernetes 集群中获取可见性需要多个界面。
* 在 Rancher 中部署多集群应用需要为每个 Rancher 服务器重复该过程。
